# Verification, review, publication, and handoff

## Candidate sequence

1. Bind each targeted verification result to exact subject identity, command semantics, scope, environment/lock identity, side effects, and invalidation rules. Query reusable evidence before every check.
2. Format changed files, run the narrowest targeted checks, and repair accepted in-scope failures. Reviewers reuse these results and run only finding-specific probes.
3. Inspect proposed paths, file types, metadata, and bytes before staging. Stage only authorized files, require the worktree bytes to match the index, and freeze the Git index tree as the immutable candidate identity.
4. Apply the review matrix to that identity. File count alone never triggers review. Multiple triggers strengthen one rubric. Phase 1 permits at most one independent read-only reviewer and no final-verifier agent.
5. If review finds an accepted in-scope defect, return to root implementation. Repair, format, run focused checks, restage, freeze a new identity, and have the same reviewer examine the new exact subject. Continue until it passes or a factual stop condition occurs; there is no repair-count limit.
6. Query reusable full-gate evidence. After review passes when required and final assembly is unchanged, the root runs `python scripts/temper-gate.py all` once for the immutable candidate. Do not rerun on the same bytes. Only recorded loss of the result can justify a replacement run; a content change creates a new candidate instead.
7. Confirm the gate did not change tracked bytes and the index tree still matches the frozen identity. Audit the exact staged diff, binary/file metadata, secrets and private-data categories, commit message, and author/committer identity.
8. Commit exactly the frozen tree and verify the commit tree equals it. Push and open or update the draft PR. A PR update triggers one workflow containing Ubuntu and Windows jobs; a later merge triggers the separate `main` push workflow. Concurrency cancels only superseded runs for the same ref.
9. If CI or review finds an accepted in-scope defect, create a new immutable revision, repeat focused repair/review as applicable, run one full gate for that new revision, audit, commit, and push. Never rerun a local gate or CI workflow merely to seek a different result on unchanged bytes.
10. Stop after both CI jobs pass and present the exact candidate for the maintainer's separate merge/integration decision.

## Evidence and handoff

Inspect public-safety classification and generated bytes whenever public output is in scope. Keep private evidence, route receipts, telemetry, and implementation reasoning private.

Write a typed checkpoint and handoff containing the task and root executor (or explicitly approved exceptional worker), exact identity, changed paths, route selection/observation/compliance/experiment label, decisions, acceptance evidence, reusable verification references, risks, stop conditions, findings, scope/safety statement, integration guidance, and exact next action.

The root-executed full-gate and public-safety results may recommend but cannot grant the separate maintainer-only `INTEGRATION_AUTHORIZED` decision.
