# Verification, review, and handoff

1. Bind each verification result to exact subject identity, command semantics, scope, environment/lock identity, side effects, and invalidation rules.
2. Query reusable verification before running a check. Reviewers run only finding-specific checks that answer a concrete unanswered question.
3. Apply the review matrix. File count alone never triggers review. Multiple triggers on one subject produce one review subject; protected boundaries receive one cold Terra-high review.
4. A second reviewer requires one concrete unresolved finding and a recorded reason that another independent perspective may resolve it. Reviewers and model perspectives do not stack automatically; unexercised premium or diversity routes remain private.
5. Inspect public-safety classification and generated bytes when public output is in scope. Keep private evidence and implementation reasoning private.
6. Write a typed checkpoint and handoff containing exact identity, decisions, evidence, verification, risks, stops, findings, and the exact next action.

One registered final-candidate verifier runs the full repository gate once after cold review and final assembly. The result may recommend but cannot grant the separate maintainer-only `INTEGRATION_AUTHORIZED` decision.
