# Temper ML v0.1 Implementation Plan

**Status:** Draft for implementation kickoff
**Date:** 2026-07-01
**Architecture source:** `docs/superpowers/specs/2026-06-30-temper-ml-architecture-design.md`
**Scope:** Bounded v0.1 walking skeleton through Noah-compatible real workflow

## 1. Purpose

This plan turns the approved Temper ML architecture direction into an executable
implementation sequence. It preserves the architecture's source-of-truth rules:
Temper owns canonical identities, immutable records, evaluation evidence, review
artifacts, registry events, and user workflow; Noah and other systems remain
adapter-backed providers rather than canonical stores.

The immediate goal is not to build every domain concept. The goal is to create a
clean-checkout, fixture-backed vertical skeleton that proves the core evidence
model, then replace fixtures with Noah-compatible real training, evaluation,
review, registry, and inference paths in controlled stages.

## 2. Non-Negotiable Constraints

Every implementation stage must satisfy these constraints:

- Do not commit secrets, private artifact identifiers, private dataset values,
  private model fingerprints, workstation paths, hostnames, process identifiers,
  usernames, IP addresses, run IDs from private systems, or private retrieval
  URIs.
- Use synthetic fixtures for public tests.
- Keep Temper domain code independent of Noah implementation imports.
- Store provider-specific data under versioned adapter namespaces.
- Treat MLflow, Noah run IDs, trainer job IDs, DVC revisions, and lab-server
  handles as external references only.
- Make identity projection versions explicit and testable before relying on
  their hashes.
- Keep the graphical UI behind application services; do not let UI state become
  canonical evidence.
- Never allow training, evaluation, review, or registry adapters to promote or
  mutate production state automatically.

## 4. Workstream A: Project Scaffolding and Cloud Contract

### Objective

Make a clean pushed checkout self-describing and runnable without private files,
secrets, GPU hardware, or agent-phase internet access.

### Implementation Tasks

1. Add pinned Python packaging and development tooling.
   - Files: `pyproject.toml`, `uv.lock`.
   - Include runtime package metadata, console script `temper`, test tooling,
     formatting/linting, and type-checking dependencies.
2. Add repository-owned cloud scripts.
   - Files: `scripts/codex-cloud-setup.sh`,
     `scripts/codex-cloud-maintenance.sh`.
   - Scripts must be idempotent, noninteractive, Linux-compatible, and avoid
     secrets.
3. Add a fixture walkthrough command.
   - File: `scripts/temper-fixture-walkthrough.sh`.
   - It must run the deterministic end-to-end fixture path from a clean checkout.
4. Update root development instructions.
   - File: `AGENTS.md`.
   - Add exact setup, lint, test, type-check, and fixture walkthrough commands.
5. Add public-safe example configuration.
   - Files: `docs/operations/configuration.md`, `.env.example` if needed.
   - Store only secret names and inert placeholders.

### Acceptance Commands

```bash
bash scripts/codex-cloud-setup.sh
bash scripts/codex-cloud-maintenance.sh
uv run pytest tests/unit tests/integration_fixture
bash scripts/temper-fixture-walkthrough.sh
```

## 5. Workstream B: Canonical JSON, Identity Projections, and Redaction

### Objective

Implement deterministic public-safe identity and diagnostics primitives before
creating domain records that depend on them.

### Implementation Tasks

1. Implement canonical JSON.
   - File: `src/temper_ml/store/canonical_json.py`.
   - Rules: UTF-8, sorted keys by Unicode code point, no insignificant
     whitespace, one trailing newline, finite integers and explicitly normalized
     decimals only, duplicate-key rejection on read, no implicit Unicode
     normalization.
2. Implement projection hashing.
   - Files: `src/temper_ml/domain/projections.py`,
     `schemas/identity-projections/*.schema.json`.
   - Hash form: `SHA-256("temper:" + projection-name-and-version + "\n" +
     canonical-json(projected-fields))`.
3. Implement byte and bundle hashing.
   - Files: `src/temper_ml/domain/artifacts.py`,
     `src/temper_ml/store/verifier.py`.
   - Reject absolute paths, traversal, symlinks, duplicate normalized member
     paths, and mutable path-derived identity.
4. Implement public-safe redaction.
   - File: `src/temper_ml/store/redaction.py`.
   - Default exports must redact local usernames, absolute paths, hostnames,
     process IDs, IP addresses, MAC addresses, private URLs, and provider IDs
     marked private.
5. Add fixture vectors.
   - Files: `fixtures/**/expected*.json`, `tests/unit/test_canonical_json.py`,
     `tests/unit/test_identity_projections.py`,
     `tests/unit/test_redaction.py`.

### Acceptance Commands

```bash
uv run pytest tests/unit/test_canonical_json.py tests/unit/test_identity_projections.py tests/unit/test_redaction.py
uv run pytest tests/adversarial/test_bundle_projection.py
```

## 6. Workstream C: Write-Once Store and Event Streams

### Objective

Prove immutable records, atomic writes, event-chain recovery, and derived-state
rebuild before invoking any real external backend.

### Implementation Tasks

1. Implement project layout management.
   - File: `src/temper_ml/store/layout.py`.
   - Support `.temper/` metadata paths from the architecture.
2. Implement write-once persistence.
   - File: `src/temper_ml/store/write_once.py`.
   - Write through temporary files, fsync files and containing directories where
     supported, atomically rename, reject overwrites, and verify existing hashes.
3. Implement event stream.
   - File: `src/temper_ml/store/event_stream.py`.
   - Shared for runs and registry. Include stream ID, fixed-width sequence,
     previous-event hash, event hash, idempotency key, event type/body, and
     observed time.
4. Implement recovery and verification.
   - File: `src/temper_ml/store/verifier.py`.
   - Detect missing sequence numbers, hash mismatches, idempotency conflicts,
     incomplete temporary files, invalid hash-addressed directories, and stale
     derived state.
5. Implement CLI inspection.
   - File: `src/temper_ml/cli.py`.
   - Commands: `temper status <project>`, `temper dump <project>`,
     `temper verify <project>`.

### Acceptance Commands

```bash
uv run pytest tests/unit/test_write_once.py tests/unit/test_event_stream.py
uv run pytest tests/adversarial/test_store_recovery.py
uv run temper verify fixtures/projects/minimal
uv run temper dump fixtures/projects/minimal
```

## 7. Workstream D: Fixture Walking Skeleton

### Objective

Run the smallest complete Temper workflow without Noah, MLflow, GPU hardware, or
network access.

### Implementation Tasks

1. Implement core records needed by the skeleton.
   - Files: `src/temper_ml/domain/records.py`, `projects.py`, `datasets.py`,
     `tasks.py`, `policies.py`, `experiments.py`, `runs.py`, `artifacts.py`,
     `evaluations.py`, `reviews.py`, `registry.py`.
2. Implement application services.
   - Files: `src/temper_ml/app_services/projects.py`,
     `src/temper_ml/app_services/workflow.py`.
   - Services must orchestrate domain objects and store writes without exposing
     adapter-specific details.
3. Implement fixture adapters.
   - Files: `src/temper_ml/adapters/fixture/*.py`.
   - Deterministic training creates a synthetic adapter artifact. Deterministic
     evaluation produces stable per-item results, aggregate values, and gate
     outcomes. Deterministic inference returns fixture predictions.
4. Implement fixture Eval Pack and gates.
   - File: `src/temper_ml/eval_packs/noah_fixture.py`.
5. Implement blind packet rendering and leak audit for fixtures.
   - Files: `src/temper_ml/review/packet_renderer.py`,
     `src/temper_ml/review/leak_audit.py`.
   - Produce a review payload manifest listing every reviewer-facing file/API
     payload with hash and audit result.
6. Implement registry registration event.
   - File: `src/temper_ml/domain/registry.py`.
   - Registration must not imply promotion.
7. Add CLI end-to-end command.
   - File: `src/temper_ml/cli.py`.
   - Command: `temper fixture walkthrough --project <path>`.

### Acceptance Commands

```bash
uv run pytest tests/integration_fixture/test_fixture_walkthrough.py
bash scripts/temper-fixture-walkthrough.sh
uv run temper verify .temper-fixture-output
uv run temper dump .temper-fixture-output
```

## 8. Workstream E: Noah Contract Fixtures and Adapter Schemas

### Objective

Represent the Noah compatibility boundary in Temper without requiring a Noah
checkout for ordinary development.

### Implementation Tasks

1. Define Noah adapter schemas.
   - Files: `schemas/adapters/noah/*.schema.json`,
     `src/temper_ml/adapters/noah/schemas.py`.
   - Include dataset recipe input/output, training provider request,
     evaluation provider request/result, baseline descriptor, runtime-reuse
     capability descriptor, inference request/result, and transfer receipt.
2. Add contract fixtures.
   - Files: `fixtures/noah_contracts/**`.
   - Use synthetic field names and values matching the public contract shape;
     do not include private operational IDs or fingerprints.
3. Implement Noah dataset recipe adapter shell.
   - File: `src/temper_ml/adapters/noah/dataset_recipe.py`.
   - Validate Noah-compatible fields and produce canonical dataset/task records.
4. Implement process boundaries.
   - File: `src/temper_ml/adapters/noah/process_boundary.py`.
   - Provide `LocalProcessBoundary` for tests and `WslRocmProcessBoundary` for
     capability-gated real integration.
5. Add contract tests that assert provider-specific fields remain namespaced.
   - Files: `tests/contract/test_noah_*.py`.

### Acceptance Commands

```bash
uv run pytest tests/contract/test_noah_schemas.py tests/contract/test_noah_dataset_recipe.py
uv run pytest tests/contract/test_noah_provider_requests.py
```

## 9. Workstream F: Real Noah Training Path

### Objective

Replace the fixture trainer with the versioned Noah training backend while
retaining the deterministic fixture end-to-end path.

### Implementation Tasks

1. Implement `NoahTrainingBackend`.
   - File: `src/temper_ml/adapters/noah/training_backend.py`.
   - Translate canonical training request into a fully resolved Noah provider
     request and persist `provider-request.json` before execution.
2. Capture public-safe execution evidence.
   - Record Noah public commit, adapter contract version, sanitized command,
     environment summary, backend capability version, and normalized failure.
   - Do not store private checkout paths, hostnames, PIDs, usernames, or private
     run IDs in canonical records.
3. Implement two-sided transfer receipt for WSL/ROCm output import.
   - File: `src/temper_ml/adapters/noah/process_boundary.py`.
   - Linux-side producer and Temper-side importer must compute the same declared
     artifact projection before accepting outputs.
4. Add optional MLflow tracking service.
   - File: `src/temper_ml/adapters/mlflow/tracking_service.py`.
   - Tracking failures must not invalidate canonical run success unless policy
     explicitly requires tracking.
5. Add capability-gated integration tests.
   - Files: `tests/contract/test_noah_training_backend.py`,
     `tests/integration_fixture/test_training_with_local_noah_double.py`.

### Acceptance Commands

```bash
uv run pytest tests/contract/test_noah_training_backend.py
uv run pytest tests/integration_fixture/test_training_with_local_noah_double.py
TEMPER_ENABLE_REAL_NOAH=1 uv run pytest tests/hardware/test_noah_training_rocm.py
```

The hardware command is allowed to skip with an explicit reason when the Noah
checkout, ROCm device, or private runtime descriptors are unavailable.

## 10. Workstream G: Real Evaluation, Review, and Runtime Reuse

### Objective

Integrate the verified v1 baseline, Noah Eval Pack, runtime-reuse capability,
explicit gates, leak audit, sealed judgments, reveal control, and inference
runtime.

### Implementation Tasks

1. Implement baseline resolver.
   - File: `src/temper_ml/adapters/noah/baseline_resolver.py`.
   - Resolve the policy-pinned baseline from an untracked project policy or
     approved private store, verify immutable bundle projection, and fail closed
     on mismatch.
2. Implement runtime key projection.
   - File: `src/temper_ml/runtime/keys.py`.
   - Include model/tokenizer/artifact identities, dtype, quantization,
     attention backend, device configuration, and captured software versions.
   - Exclude virtual variant ID and adapter strength from physical runtime key,
     but include both in evaluation request/variant identity.
3. Implement Noah evaluation backend.
   - File: `src/temper_ml/adapters/noah/evaluation_backend.py`.
   - Preserve caller order after physical-runtime grouping, record load counts,
     bounded peak-VRAM observations, runtime keys, raw results, and normalized
     failures.
4. Implement real review packet import and audit.
   - Files: `src/temper_ml/review/packet_renderer.py`,
     `src/temper_ml/review/leak_audit.py`.
   - Audit every reviewer-facing representation against an exact payload
     manifest before `review_open`.
5. Implement judgment sealing and reveal events.
   - File: `src/temper_ml/domain/reviews.py`.
   - Reveal must require an immutable judgment-set hash and explicit reveal
     event.
6. Implement Noah inference runtime adapter.
   - File: `src/temper_ml/adapters/noah/inference_runtime.py`.
   - Connect through local HTTP contract and treat lab-server handles as
     external references.

### Acceptance Commands

```bash
uv run pytest tests/contract/test_runtime_keys.py tests/contract/test_noah_evaluation_backend.py
uv run pytest tests/integration_fixture/test_review_lifecycle.py
TEMPER_ENABLE_REAL_NOAH=1 uv run pytest tests/hardware/test_noah_evaluation_runtime_reuse.py
TEMPER_ENABLE_REAL_NOAH=1 uv run pytest tests/hardware/test_noah_inference_runtime.py
```

Hardware commands are capability-gated and must skip with explicit reasons when
requirements are unavailable.

## 11. Workstream H: Graphical Workflow

### Objective

Build the local browser application over already-proven application services.
The UI should expose one coherent Temper workflow rather than external-tool
navigation.

### Implementation Tasks

1. Implement local loopback server.
   - File: `src/temper_ml/ui/server.py`.
   - It must call application services and never write canonical evidence
     directly.
2. Implement project workflow pages.
   - Create/open project, import dataset, configure experiment, launch run,
     inspect status/logs, compare evaluation, open review, register artifact,
     test inference, and display recommendation.
3. Add accessibility and error-state tests.
   - Files: `tests/ui/**`.
4. Add screenshot-based manual verification only after visible UI changes.

### Acceptance Commands

```bash
uv run pytest tests/ui
uv run temper ui --project .temper-fixture-output --host 127.0.0.1 --port 8765
```

## 12. Cross-Repository Noah Handoffs

Noah changes must be coordinated through explicit handoff records rather than
implicit local assumptions.

Each Noah handoff must include:

- public Noah base commit;
- Temper adapter contract version;
- expected command/file/local-HTTP contract;
- synthetic fixture input and output shape;
- acceptance criteria;
- public-safe evidence required from real hardware runs;
- private descriptors required at runtime, named by secret/config key only.

Proposed handoff files:

```text
docs/handoffs/noah-training-backend-v1.md
docs/handoffs/noah-evaluation-runtime-reuse-v1.md
docs/handoffs/noah-inference-runtime-v1.md
```

## 13. Stage Gates

Implementation should advance only when the previous gate remains green.

| Gate | Required evidence |
| --- | --- |
| Gate 1: Scaffolding | setup, maintenance, lint, type-check, unit tests run from clean checkout |
| Gate 2: Identity/store | canonical JSON vectors, projection vectors, write-once tests, recovery tests |
| Gate 3: Fixture skeleton | deterministic fixture walkthrough, verify, dump, no network/secrets/GPU |
| Gate 4: Noah contracts | schema and adapter contract tests pass without Noah checkout |
| Gate 5: Real training | capability-gated Noah training evidence attached from capable environment |
| Gate 6: Real evaluation/review | baseline verification, runtime reuse evidence, leak audit, sealed judgments |
| Gate 7: UI hardening | UI tests pass and workflow uses application services only |

## 14. Definition of Done for v0.1

v0.1 is done when:

- a clean checkout can run setup and deterministic fixture walkthrough without
  secrets, private files, GPU hardware, or agent-phase internet access;
- every v1 content identity has a named projection and test vector;
- `temper status`, `temper dump`, and `temper verify` explain canonical records
  before the UI is required;
- an interrupted or failed run leaves hash-linked events and normalized failure
  evidence;
- Noah-specific fields and thresholds are confined to Noah adapters, Eval Pack,
  task definition, recipe, and project policy;
- optional MLflow references never become canonical identity;
- the baseline resolves to verified immutable bytes and configured strength
  without consulting mutable registry state;
- blind review cannot open before mechanical leak audit passes for every
  reviewer-facing payload;
- reveal cannot occur before judgments are sealed;
- registration does not imply promotion;
- UI workflow presents Temper as one coherent local application.

## 15. First Pull Request Recommendation

The first implementation PR should be intentionally small:

1. Add `pyproject.toml`, pinned tooling, and empty package layout.
2. Add canonical JSON and projection hashing with test vectors.
3. Add write-once store primitives with unit tests.
4. Add setup/test commands to `AGENTS.md`.
5. Do not add Noah adapters, UI, MLflow, or hardware integration yet.

This keeps the first executable change focused on the correctness substrate that
all later workflow stages depend on.
