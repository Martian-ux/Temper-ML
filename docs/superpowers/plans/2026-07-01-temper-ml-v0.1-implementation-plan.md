# Temper ML v1 Implementation Plan

**Status:** Historical v0.1 planning reference; superseded for execution sequencing
**Date:** 2026-07-09
**Architecture source:** docs/superpowers/specs/2026-06-30-temper-ml-architecture-design.md
**Scope:** A local-first LLM adapter experimentation product with a Temper-owned runtime.

The path remains published for stable links and implementation history. Do not
use its workstream order to start or schedule v1 work. The adopted governing
execution sequence is the July 9 roadmap at
`docs/superpowers/plans/2026-07-09-temper-ml-v1-execution-roadmap.md`.

## 1. Purpose

This plan turns the v1 architecture into an executable, acceptance-tested
sequence. Temper owns the supported workflow for task-centered adapter projects:
dataset preparation, recipe resolution, run lifecycle, evaluation, review,
recommendation, retention, replay, bounded iteration, merge candidates, and
readiness assessment.

The plan deliberately replaces the previous provider-first roadmap. Mature
libraries power the normal training path, but no external trainer product is the
primary v1 runtime or the source of truth. Compatibility engines and Noah
integration are optional, later boundaries.

## 2. Non-Negotiable Constraints

Every workstream must preserve these rules:

- Temper is limited to LLM adapter experimentation, not general ML.
- Temper records are authoritative for project, dataset, experiment, run,
  artifact, metrics, registry, retention, and recommendation state.
- UI and CLI call application services; neither writes canonical records
  directly.
- Canonical records are immutable, hash-addressed, and public-safe. Derived
  state and caches are rebuildable.
- An experiment is immutable intent; a run is one execution attempt. Material
  configuration changes create derived experiments.
- Normal product workflows do not require an external dashboard.
- v1 uses no model judges and no universal quality score.
- Human evidence, automated evidence, user decisions, and overrides stay
  distinct.
- All loops have explicit resource and stopping bounds.
- Cleanup deletes heavy bytes only through an audited lifecycle event. It never
  deletes ordinary canonical evidence.
- No workflow automatically mutates production or deploys an artifact.
- Public source, fixtures, tests, documents, and commit metadata contain no
  credentials, private data, private identifiers, local paths, or operational
  details.

## 3. Delivery Strategy

Deliver one coherent vertical slice before broadening behavior:

1. Make canonical storage and v1 domain contracts trustworthy.
2. Prove a deterministic dataset-to-adapter fixture path through the
   Temper-owned runtime.
3. Add integrity-first evaluation, recommendation evidence, and a focused UI.
4. Replace only the fixture training internals with library-backed local
   execution while preserving the same manifest and evidence contracts.
5. Add replay, retention, loops, merge candidates, and readiness only after the
   single-run path is proven.
6. Add Noah and other external systems as compatibility boundaries only when
   they are justified by a supported workflow.

Each stage must have synthetic fixtures, contract tests, and a clear gate before
the next stage begins.

## 5. Workstream A: Foundation and Canonical Evidence

### Objective

Finish the public-safe foundation that all later records depend on. The existing
package, canonical JSON, content identities, and write-once store are the
starting point; this workstream completes their v1 contract.

### Tasks

1. Complete canonical JSON, projection, byte, and bundle verification.
   - Enforce canonical bytes, strict UTF-8, duplicate-key rejection, stable
     projection versions, and symlink-safe verification.
2. Implement canonical store layout and append-only event streams.
   - Rebuild derived state from hash-linked events.
   - Ignore incomplete temporary files and fail closed on identity mismatch.
3. Implement public-safe redaction and export boundaries.
4. Add inspection commands for status, dump, verify, and manifest display.
5. Add synthetic identity, corruption, recovery, redaction, and path-safety
   fixtures.

### Gate

    uv run pytest tests/unit/test_canonical_json.py tests/unit/test_identity_projections.py
    uv run pytest tests/unit/test_write_once.py tests/unit/test_event_stream.py
    uv run pytest tests/adversarial/test_store_recovery.py tests/adversarial/test_bundle_projection.py
    uv run temper verify fixtures/projects/minimal

## 6. Workstream B: Task-Centered Project and Experiment Contracts

### Objective

Define the immutable records that make Temper an adapter-experiment product
rather than a general lab folder.

### Tasks

1. Define Project, TaskDefinition, ProjectPolicy, BaseModelRevision, and
   CompatibilityGroup schemas.
   - A project binds one task, evaluation policy, confirmation suites,
     retention policy, readiness policy, recipe families, and baseline policy.
2. Define explicit compatibility relationships.
   - Comparable, merge-compatible, resume-compatible, and
     deployment-compatible relationships must be independently testable.
3. Define versioned Recipe and RecipeResolution records.
   - Capture user-facing choices and all resolved technical settings.
4. Define HardwareRequirements, HardwareCapabilityProfile, preflight estimate,
   and resolution-diff records.
5. Define Experiment, DerivedExperiment, Run, and ReplayPlan records.
   - Strict replay preserves the original manifest.
   - Assisted adaptation creates a visibly derived experiment.
6. Define Artifact, ArtifactAvailability, lineage, and parent relationships.
7. Define baseline records for per-model baseline, project champion, and fixed
   reference baseline.

### Gate

    uv run pytest tests/contract/test_project_task_contract.py
    uv run pytest tests/contract/test_compatibility_groups.py
    uv run pytest tests/contract/test_recipe_resolution.py
    uv run pytest tests/contract/test_experiment_and_replay_lineage.py

## 7. Workstream C: Deterministic LLM Dataset Pipeline

### Objective

Build the narrow dataset path required for adapter training, with no general
data-editor ambitions.

### Tasks

1. Implement import adapters for JSON, JSONL, CSV, and supported Hugging Face
   datasets.
2. Implement explicit field mapping and task rendering contracts.
3. Validate records, filter invalid rows, record exclusion reasons, and apply
   deterministic deduplication.
4. Capture tokenizer identity, token-length analysis, deterministic splits,
   rendered training text, previews, and summary statistics.
5. Produce correction reports and reimport comparisons without editing the
   authoritative source data.
6. Add public synthetic fixtures for valid, invalid, duplicate, long-context,
   and reimported data.

### Gate

    uv run pytest tests/unit/test_dataset_pipeline.py
    uv run pytest tests/contract/test_dataset_import_contract.py
    uv run pytest tests/integration_fixture/test_dataset_reimport.py

## 8. Workstream D: Temper-Owned Fixture Runtime

### Objective

Prove the normal product path without a GPU, network, external trainer, or
external dashboard.

### Tasks

1. Implement runtime preflight and recipe-resolution services.
2. Implement a deterministic fixture adapter-training runtime that consumes the
   same resolved manifest the real runtime will consume.
3. Record launch, progress, checkpoint, cancellation, interruption, recovery,
   artifact-ingestion, and terminal events through Temper services.
4. Implement artifact integrity validation independently of quality evaluation.
5. Add fixture inference and base-model compatibility checks.
6. Add a command that creates a project, imports data, resolves a recipe,
   freezes an experiment, executes a run, ingests an artifact, and verifies it.

### Gate

    uv run pytest tests/integration_fixture/test_native_runtime_walkthrough.py
    uv run pytest tests/contract/test_runtime_lifecycle.py
    uv run temper fixture walkthrough --project .temper-fixture-output
    uv run temper verify .temper-fixture-output

## 9. Workstream E: Evaluation, Review, Recommendations, and Playground

### Objective

Make evidence quality explicit before building optimization or deployment-readiness
features.

### Tasks

1. Implement evaluation modes: no quality evaluation, light evaluation, full
   suite, and experiment loop.
2. Implement artifact-integrity validation for every mode.
3. Implement development, regression, and confirmation case suites.
   - Record soft-seal states: sealed, unsealed, modified, contaminated, and
     retired.
4. Implement deterministic checks, held-out loss, task metrics, format checks,
   and explicit baseline comparisons.
5. Prohibit model-judge evaluators in v1 schemas and runtime validation.
6. Implement structured solo review records and optional blind-review packets.
   - Blind review uses a leak audit and sealed-judgment reveal controls.
7. Implement a policy-based recommendation engine.
   - Support hard qualifiers, advisory metrics, objectives, conflicts,
     lexicographic ranking, evidence confidence, and user override decisions.
8. Implement the playground service with synchronized prompts, comparison,
   inference controls, saved outputs, notes, ratings, replay, and case capture.

### Gate

    uv run pytest tests/unit/test_evaluation_policy.py tests/unit/test_recommendations.py
    uv run pytest tests/contract/test_soft_seal_states.py
    uv run pytest tests/integration_fixture/test_playground_review.py
    uv run pytest tests/integration_fixture/test_optional_blind_review.py

## 10. Workstream F: Early Product UI and CLI

### Objective

Expose the proven services through one coherent local product interface.

### Tasks

1. Implement a loopback UI server, routes, and views over application services.
2. Implement project setup, dataset import, recipe selection and resolution
   preview, hardware preflight, run control, status, and logs.
3. Implement evaluation and playground views with side-by-side comparison,
   hidden identities when requested, notes, ratings, and recommendation state.
4. Implement artifact inventory, registry decisions, and manifest inspection.
5. Implement the matching CLI commands for status, dump, verify, preflight,
   replay planning, and fixture workflows.
6. Add UI tests proving routes cannot bypass application services for canonical
   writes.

### Gate

    uv run pytest tests/ui tests/integration_fixture/test_ui_vertical_slice.py
    uv run temper ui --project .temper-fixture-output --host 127.0.0.1 --port 8765

## 11. Workstream G: Library-Backed Local Adapter Runtime

### Objective

Replace fixture training internals with a Temper-controlled local adapter runtime
built from mature libraries, without changing project or manifest semantics.

### Tasks

1. Integrate the approved library layer for tokenization, PEFT adapters,
   training, acceleration, quantization, and checkpointing.
2. Translate a resolved Temper manifest into the exact library configuration
   without creating an external-provider manifest as the primary record.
3. Capture library versions, model and tokenizer revisions, preflight estimate,
   device identity, metrics, checkpoints, logs, cancellation, and recovery
   evidence.
4. Enforce recipe and hardware constraints before launch.
5. Add capability-gated hardware tests and deterministic local doubles.
6. Verify that a real run, fixture run, and replay use the same domain and
   storage contracts.

External trainer engines are not part of this gate. Any later compatibility
backend must prove that it preserves these contracts.

### Gate

    uv run pytest tests/contract/test_native_adapter_runtime.py
    uv run pytest tests/integration_fixture/test_library_runtime_double.py
    TEMPER_ENABLE_LOCAL_TRAINING=1 uv run pytest tests/hardware/test_local_adapter_runtime.py

## 12. Workstream H: Retention, Cleanup, and Reproduction

### Objective

Make local storage costs and reproduction semantics honest and safe.

### Tasks

1. Implement full-retention defaults and artifact-availability records.
2. Build a cleanup planner that calculates shared-byte savings and losses to
   resumability, inspectability, final-adapter availability, cache convenience,
   and debugging evidence.
3. Require an immutable cleanup receipt and lifecycle event for heavy-byte
   deletion.
4. Preserve canonical manifests, hashes, events, final metrics,
   recommendations, and cleanup receipts during ordinary cleanup.
5. Implement strict replay planning and assisted adapted-replay derivation.
6. Add UI and CLI views for inventory, cleanup preview, cleanup result, and
   strict-versus-adapted replay comparison.

### Gate

    uv run pytest tests/unit/test_retention_planner.py tests/unit/test_replay_plans.py
    uv run pytest tests/integration_fixture/test_cleanup_lifecycle.py
    uv run pytest tests/adversarial/test_canonical_evidence_survives_cleanup.py

## 13. Workstream I: Bounded Loops, LoRA Merging, and Readiness

### Objective

Add reproducible iteration without turning Temper into an unbounded optimizer or
deployment platform.

### Tasks

1. Implement manual next-trial selection and optimized loop policies.
   - Require search space, hard qualifiers, objectives, budgets, and stopping
     conditions before automated execution.
2. Enforce limits on trial count, wall time, GPU time, disk budget, or explicit
   stopping criteria.
3. Evaluate loop-selected candidates against a separate confirmation suite.
4. Implement Pareto views for qualified candidates with real tradeoffs.
5. Implement LoRA merge planning and execution.
   - Require same base-model and tokenizer revisions, compatible target
     modules, integrity-verified parents, a maintained merge method, immutable
     parent lineage, and comparison against every parent.
6. Implement runtime-target readiness policies and assessments.
   - Ready, approved by user, not ready, and unevaluated are evidence-aware
     labels; no production runtime is controlled.

### Gate

    uv run pytest tests/unit/test_loop_policy.py tests/unit/test_merge_compatibility.py
    uv run pytest tests/contract/test_readiness_policy.py
    uv run pytest tests/integration_fixture/test_bounded_loop.py
    uv run pytest tests/integration_fixture/test_lora_merge_lineage.py

## 14. Workstream J: Optional Compatibility Boundaries

### Objective

Add Noah or an external training engine only when it fills a measured,
supported-product gap without weakening Temper ownership.

### Tasks

1. Define versioned compatibility contracts and public synthetic fixtures.
2. Translate Temper manifests into namespaced compatibility requests and ingest
   returned evidence as external references.
3. Verify that external IDs, logs, metrics, files, and outputs never replace
   Temper identities or lifecycle state.
4. Keep Noah-specific fields, thresholds, and data recipes entirely inside the
   Noah compatibility package.
5. Add compatibility tests that prove the native runtime remains the default
   path and that normal UI navigation does not require the external system.

There is no provider bake-off gate for core v1 delivery. A compatibility engine
may be evaluated later against explicit capability, maintenance, and evidence
criteria.

### Gate

    uv run pytest tests/contract/test_compatibility_boundary.py
    uv run pytest tests/contract/test_noah_isolation.py
    TEMPER_ENABLE_COMPATIBILITY_RUNTIME=1 uv run pytest tests/hardware/test_compatibility_ingestion.py

## 15. Stage Gates

Implementation advances only when the previous gate remains green.

| Gate | Required evidence |
| --- | --- |
| 1: Foundation | Canonical records, store recovery, verification, redaction, and public-safe fixtures |
| 2: Product contracts | Task-centered project, compatibility, recipes, hardware, experiments, and baseline contracts |
| 3: Dataset | Deterministic import, rendering, exclusions, splits, and reimport comparison |
| 4: Fixture runtime | Manifest-to-artifact run lifecycle, integrity validation, and recovery evidence |
| 5: Evaluation | Case suites, soft sealing, recommendation policy, structured review, and playground evidence |
| 6: Product UI | UI uses services only and completes the fixture vertical slice |
| 7: Local runtime | Library-backed local adapter training preserves Temper contracts |
| 8: Retention and replay | Cleanup receipts, availability state, strict replay, and adapted derivation |
| 9: Iteration | Bounded loops, merge lineage, confirmation evaluation, and readiness assessment |
| 10: Compatibility | Optional external or Noah boundary preserves Temper ownership |

## 16. Definition of Done for v1

v1 is done when a clean checkout can run a supported local adapter project that:

- imports and freezes a deterministic LLM dataset;
- resolves a versioned recipe and shows its hardware-aware changes before launch;
- creates immutable experiments and separate run attempts;
- trains through the Temper-owned local runtime and verifies the resulting
  artifact even without quality evaluation;
- evaluates per-model, champion, and fixed-reference baselines where applicable;
- records deterministic and human evidence without model judges;
- supports structured playground review, optional blind review, soft-sealed
  confirmation suites, policy recommendations, and user overrides;
- retains heavy bytes by default and performs audited cleanup safely;
- distinguishes strict replay from hardware-adapted derived experiments;
- runs bounded experiment loops and evaluates compatible merged LoRA candidates;
- records deployment readiness without managing deployment; and
- remains usable without external dashboards, external trainers, cloud accounts,
  or Noah.

Every core record must be inspectable through the UI and CLI and trace to
immutable identities. All public tests use synthetic data and all hardware tests
are capability-gated with explicit skip reasons.
