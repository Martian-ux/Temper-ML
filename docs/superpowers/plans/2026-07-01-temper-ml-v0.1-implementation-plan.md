# Temper ML v0.1 Implementation Plan

**Status:** Draft for implementation kickoff, amended for unified local-first product direction
**Date:** 2026-07-01
**Architecture source:** `docs/superpowers/specs/2026-06-30-temper-ml-architecture-design.md`
**Scope:** Bounded v0.1 unified Temper workflow, fixture-first UI, Noah compatibility, measured training-provider selection

## 1. Purpose

This plan turns the Temper ML architecture into an executable sequence for a
unified, local-first model experimentation product. Temper owns projects,
dataset identity, experiment and training configuration, run history, logs,
metrics, charts, candidate comparison, evaluation suites, gates,
recommendations, blind review, artifact registration, local execution controls,
interruption and retry evidence, basic sweeps, and all supported navigation.

The immediate goal is not to build every domain concept. The goal is to create a
clean-checkout, fixture-backed vertical skeleton with an early browser UI, then
replace fixtures with Noah-compatible real training, evaluation, review,
registry, and inference paths in controlled stages. Normal users must not need
to open MLflow, Optuna Dashboard, Argilla, DVC Studio, Axolotl, LLaMA Factory,
or another external interface as part of the supported Temper workflow.

## 2. Non-Negotiable Constraints

Every implementation stage must satisfy these constraints:

- Do not commit secrets, private artifact identifiers, private dataset values,
  private model fingerprints, workstation paths, hostnames, process identifiers,
  usernames, IP addresses, run IDs from private systems, or private retrieval
  URIs.
- Use synthetic fixtures for public tests.
- Keep Temper domain code independent of Noah implementation imports.
- Keep provider-specific data under versioned adapter namespaces.
- Treat MLflow, Noah run IDs, trainer job IDs, DVC revisions, lab-server
  handles, and optimization-study IDs as external references only.
- Make identity projection versions explicit and testable before relying on
  their hashes.
- Keep the UI behind application services; UI code must never write canonical
  records directly.
- Keep MLflow optional and subordinate; Temper must remain usable while MLflow
  is stopped or unavailable.
- Do not expose or embed external dashboards as supported product workflow.
- Never allow training, evaluation, review, registry, sweep, or tracking
  adapters to promote or mutate production state automatically.
- Do not create speculative modules merely because they appear in a proposed
  repository tree; every file and abstraction must support an acceptance-tested
  behavior in the current stage.

## 4. Workstream A: Repository Scaffolding and Public-Safe Setup

### Objective

Make a clean pushed checkout self-describing and runnable without private files,
secrets, GPU hardware, external dashboards, or agent-phase internet access.

### Implementation Tasks

1. Add pinned Python packaging and development tooling.
   - Files: `pyproject.toml`, `uv.lock`.
   - Include package metadata, console script `temper`, test tooling,
     formatting/linting, and type-checking dependencies.
2. Add repository-owned cloud scripts.
   - Files: `scripts/codex-cloud-setup.sh`,
     `scripts/codex-cloud-maintenance.sh`.
   - Scripts must be idempotent, noninteractive, Linux-compatible, and avoid
     secrets.
3. Add a fixture walkthrough command shell.
   - File: `scripts/temper-fixture-walkthrough.sh`.
   - It must eventually run the deterministic fixture product path from a clean
     checkout.
4. Update root development instructions.
   - File: `AGENTS.md`.
   - Add exact setup, lint, test, type-check, fixture walkthrough, and public
     safety commands.
5. Add public-safe example configuration.
   - Files: `docs/operations/configuration.md`, `.env.example` if needed.
   - Store only secret names and inert placeholders.

### Acceptance Commands

```bash
bash scripts/codex-cloud-setup.sh
bash scripts/codex-cloud-maintenance.sh
uv run pytest tests/unit
bash scripts/temper-fixture-walkthrough.sh --help
```

## 5. Workstream B: Canonical Identity and Persistence Primitives

### Objective

Implement deterministic public-safe identity, redaction, write-once records,
event streams, and diagnostics before building product workflows that depend on
them.

### Implementation Tasks

1. Implement canonical JSON.
   - File: `src/temper_ml/store/canonical_json.py`.
   - Rules: UTF-8, sorted keys by Unicode code point, no insignificant
     whitespace, one trailing newline, finite integers and explicitly normalized
     decimals only, duplicate-key rejection on read, no implicit Unicode
     normalization.
2. Implement projection hashing.
   - Files: `src/temper_ml/domain/projections.py`,
     `schemas/identity-projections/*.schema.json`.
   - Hash form: `SHA-256("temper:" + projection-name-and-version + "\n" +
     canonical-json(projected-fields))`.
3. Implement byte and bundle hashing.
   - Files: `src/temper_ml/domain/artifacts.py`,
     `src/temper_ml/store/verifier.py`.
   - Reject absolute paths, traversal, symlinks, duplicate normalized member
     paths, and mutable path-derived identity.
4. Implement write-once persistence and event streams.
   - Files: `src/temper_ml/store/write_once.py`,
     `src/temper_ml/store/layout.py`, `src/temper_ml/store/event_stream.py`.
   - Write atomically, reject overwrites, verify existing hashes, rebuild
     derived state from hash-linked events, and ignore incomplete temporary
     files.
5. Implement public-safe redaction.
   - File: `src/temper_ml/store/redaction.py`.
   - Default exports must redact local usernames, absolute paths, hostnames,
     process IDs, IP addresses, MAC addresses, private URLs, and provider IDs
     marked private.
6. Add CLI inspection.
   - File: `src/temper_ml/cli.py`.
   - Commands: `temper status <project>`, `temper dump <project>`,
     `temper verify <project>`.

### Acceptance Commands

```bash
uv run pytest tests/unit/test_canonical_json.py tests/unit/test_identity_projections.py tests/unit/test_redaction.py
uv run pytest tests/unit/test_write_once.py tests/unit/test_event_stream.py
uv run pytest tests/adversarial/test_bundle_projection.py tests/adversarial/test_store_recovery.py
uv run temper verify fixtures/projects/minimal
uv run temper dump fixtures/projects/minimal
```

## 9. Workstream F: Backend Bake-Off and Parity Testing

### Objective

Select the long-term default real training provider from measured evidence, not
advertised benchmarks or the fact that Noah's current trainer already works.
Noah's current trainer remains fallback and parity reference unless another
provider clearly performs better.

### Implementation Tasks

1. Create one frozen workload and equivalent training contract.
   - Files: `fixtures/bakeoff/frozen_workload/*`,
     `src/temper_ml/adapters/providers/bakeoff.py`,
     `docs/handoffs/training-backend-bakeoff-v1.md`.
2. Compare at minimum:
   - existing Noah Transformers/PEFT trainer;
   - Axolotl;
   - Unsloth Core, if the current AMD/ROCm environment supports the required
     workflow reliably;
   - LLaMA Factory only if setup and parity requirements remain reasonably
     bounded.
3. Control, as closely as providers permit: dataset snapshot, train/validation
   split, rendered training text, tokenizer revision, chat template, sequence
   length, LoRA target modules, LoRA rank, LoRA alpha, LoRA dropout,
   quantization mode, optimizer, learning-rate schedule, effective batch size,
   gradient accumulation, seed, number of optimizer steps, evaluation
   frequency, checkpoint frequency, base-model revision, and precision
   settings.
4. Measure peak VRAM, host RAM, swap usage, tokens or samples per second, setup
   and model-load time, total training wall time, training stability,
   validation loss, final Temper evaluation results, blind-review results when
   practical, adapter compatibility, reproducibility across repeated runs,
   cancellation behavior, recovery behavior, log/evidence quality, integration
   difficulty, and provider-specific code burden.
5. Record a provider selection decision with fallback criteria and maintenance
   burden.

### Acceptance Commands

```bash
uv run pytest tests/contract/test_bakeoff_contract.py
TEMPER_ENABLE_BACKEND_BAKEOFF=1 uv run pytest tests/hardware/test_training_backend_bakeoff.py
```

The hardware command is capability-gated and may skip with an explicit reason
when ROCm devices, provider installs, public model access, or private runtime
descriptors are unavailable.

## 15. Deferred Integrations and Future Compact Evaluator Models

Additional providers or integrations are introduced only after a supported
product slice demonstrates need. For each external tool, architecture review
must identify the likely reuse mode: library, headless engine, optional
compatibility adapter, runtime dependency, storage provider, narrow extraction,
or full fork as a last resort. The plan must not expose or embed external
dashboards as normal workflow.

Compact task-specific model training is preserved for later and remains outside
v0.1. Future terms should be **Compact Evaluator Models**, **Task-Specific
Models**, or **Evaluator Studio**, not TinyML unless targeting microcontrollers
or similarly constrained embedded systems. Future compact evaluators require
explicit task definitions, versioned datasets, held-out evaluation, calibration
evidence, declared limitations, maturity status, versioned artifact identity,
and clear disclosure when used in an evaluation suite.

## 17. Stage Gates

Implementation should advance only when the previous gate remains green.

| Gate | Required evidence |
| --- | --- |
| Gate 1: Scaffolding | setup, maintenance, lint/type/unit checks run from clean checkout |
| Gate 2: Identity/store | canonical JSON vectors, projection vectors, write-once tests, recovery tests, redaction tests |
| Gate 3: Fixture skeleton | deterministic fixture walkthrough, verify, dump, no network/secrets/GPU |
| Gate 4: Early UI | fixture-backed browser workflow uses application services only |
| Gate 5: Training contract | provider contract, evidence schema, cancellation/recovery capabilities, namespacing tests |
| Gate 6: Backend bake-off | frozen workload, controlled comparison, measured selection or documented fallback |
| Gate 7: Noah contracts | schema and adapter contract tests pass without Noah checkout |
| Gate 8: Real training | selected provider evidence attached from capable environment; Noah fallback remains parity reference |
| Gate 9: Real evaluation/review/inference | baseline verification, evaluation evidence, leak audit, sealed judgments, inference contract |
| Gate 10: UI hardening | real workflow remains in Temper UI with native metrics/charts/errors |
| Gate 11: Basic sweeps | native trial queue/status/cancellation/comparison over normal Temper evaluation |

## 18. Definition of Done for v0.1

v0.1 is done when:

- a clean checkout can run setup and deterministic fixture walkthrough without
  secrets, private files, GPU hardware, external dashboards, or agent-phase
  internet access;
- every v1 content identity has a named projection and test vector;
- `temper status`, `temper dump`, and `temper verify` explain canonical records
  before and alongside the UI;
- the minimal browser UI appears before real hardware integrations are complete
  and uses application services for all canonical writes;
- an interrupted or failed run leaves hash-linked events and normalized failure
  evidence;
- Noah remains the first compatibility workflow, with dataset recipe, task
  definition, Eval Pack, baseline policy, review behavior, artifact policy, and
  recommendation policy separated from the selected training provider;
- the selected default real training provider is justified by measured bake-off
  evidence, or Noah's current trainer is explicitly retained as fallback and
  parity reference;
- optional MLflow mirroring never becomes canonical identity and Temper remains
  fully usable while MLflow is stopped;
- native run history, metrics, charts, candidate comparison, review, registry,
  and basic sweeps support the vertical slice without external dashboards;
- blind review cannot open before mechanical leak audit passes for every
  reviewer-facing payload;
- reveal cannot occur before judgments are sealed;
- registration does not imply promotion;
- compact task-specific model training is documented as future work outside
  v0.1;
- no section or behavior implies complete parity with external tools;
- no supported operation requires users to leave Temper for normal operation.

## 19. First Pull Request Recommendation

The first implementation PR should be intentionally small:

1. Add `pyproject.toml`, pinned tooling, and package skeleton.
2. Add canonical JSON and projection hashing with test vectors.
3. Add write-once store primitives with unit tests.
4. Add setup/test commands to `AGENTS.md`.
